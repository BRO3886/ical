# Journal Entry 005 - 2026-02-24 - Timezone display fix

## Goal

Fix `ical list` / `ical today` showing times in the event's stored timezone instead of the user's local timezone. Events created in ET were displaying in ET even for IST users. Additionally, `ical show` should display both local time AND the event's original timezone for cross-timezone visibility.

## What Changed

### `internal/ui/output.go`

- **`localizeTime(t, tz)`**: Simplified — now always returns `t.In(time.Local)`, ignoring the `tz` argument entirely (marked `_`). This fixes `list`/`today`/`upcoming`: all times now display in user's local timezone.

- **New `localizeTimeInZone(t, tz, referenceLoc)`**: Returns `*time.Time` in the event's own timezone, but only when it differs from `referenceLoc` by UTC offset. Returns `nil` when: tz is empty, tz is invalid, or event tz offset == reference offset (avoids redundant `(IST)` next to IST times). Takes `referenceLoc` as a parameter (not hardcoded `time.Local`) for testability.

- **`printEventDetailTable`**: Start/End lines now show dual time when `localizeTimeInZone` returns non-nil:
  ```
  Start:  Tue, 24 Feb 2026 22:30 IST  (11:00 CST)
  End:    Tue, 24 Feb 2026 23:30 IST  (12:00 CST)
  ```

- **`printEventDetailPlain`**: Same dual-time pattern using RFC3339 format.

### `internal/ui/output_test.go`

Rewrote timezone tests to be fully timezone-agnostic — no `time.Local` dependency, no skips based on machine timezone. Tests use fixed `*time.Location` values as both reference and event timezone:

- `TestLocalizeTime`: Verifies `localizeTime` always returns `time.Local` regardless of tz arg.
- `TestLocalizeTimeInZone`: 10 sub-tests covering IST↔EST, IST↔CST, EST↔IST, EST↔CST, GMT↔IST, CST↔EST, plus nil cases (empty tz, invalid tz, same-offset pairs). All wall-clock times verified exactly.

## Key Insights

- **EventKit `TimeZone` field is not reliable as the "creation timezone"**: The field reflects what macOS Calendar stored, which may already be converted. E.g., "Feeble Team Sync" created by a Google Calendar invite has `TimeZone=Asia/Kolkata` even though the organizer is in ET. The dual-display is still useful for events genuinely stored in foreign timezones (e.g., `America/Chicago` meetings on an IST machine).

- **Tests must not use `time.Local`**: The old `TestLocalizeTime` used `time.Local` for both reference and assertion, meaning tests would skip or fail differently on IST vs ET machines. Fix: pass explicit `*time.Location` to `localizeTimeInZone` so tests are deterministic everywhere.

- **Offset comparison, not name comparison**: Two timezones can have the same UTC offset (e.g., UTC and GMT, or EST and a fixed `-05:00`). Using `_, offset := t.In(loc).Zone()` (integer seconds) correctly handles this — name-based comparison would show redundant suffixes.

## Decisions Made

- **`localizeTimeInZone` takes `referenceLoc *time.Location`** instead of hardcoding `time.Local`. Slight API verbosity at call sites (`time.Local` always passed in production) but makes the function purely deterministic and testable without env manipulation.

- **`list`/`today` show only local time** (no dual display) — keeps the table compact. Dual display only in `show` detail view where space isn't constrained.

- **`nil` return from `localizeTimeInZone`** (not a boolean flag) — caller gets the converted `time.Time` directly if non-nil, avoiding a second conversion call.
