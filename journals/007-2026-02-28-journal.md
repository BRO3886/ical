# Journal Entry 007 - 2026-02-28 - OpenClaw agent support

## Goal

Add OpenClaw as a third supported agent target for `ical skills install/uninstall`, alongside the existing `claude` and `codex` targets.

## What Changed

- **`internal/skills/skills.go`**: Added `OpenClaw` entry to `DefaultTargets()` with key `"openclaw"` and `~/.openclaw/skills` base dir
- **`cmd/ical/commands/skills.go`**: Updated `--agent` flag description and error message in `resolveAgentFlag()` to include `openclaw` as a valid value; updated install command `Long` help text to show openclaw path
- **`internal/skills/skills_test.go`**: Updated `TestDefaultTargets` count assertion from 2→3, added openclaw target assertions; added `TestDetectAgentsAllPresent` (all 3 dirs present → 3 detected) and `TestDetectAgentsOpenclawOnly` (only openclaw dir → 1 detected)

## Key Insights

- `DetectAgents()` is fully generic — it just stats `filepath.Dir(t.BaseDir)` for each target. No hardcoded agent logic. Adding a new target to `DefaultTargets()` automatically gets detection for free.
- `resolveAgentFlag()` iterates `allTargets` by key match, so the `--agent openclaw` flag just works once the target is in `DefaultTargets()`. The only manual update needed was the error message listing valid values.
- `TestDetectAgentsBothPresent` was left as-is (still valid: creates 2 dirs, expects 2) — added `TestDetectAgentsAllPresent` as the new "all targets" case rather than modifying the existing test, which would change its semantics.

## Decisions Made

- Installed path `~/.openclaw/skills/ical-cli/` — follows the same convention as `~/.claude/skills/` and `~/.agents/skills/`
- No changes to `SKILL.md` or embedded skill content — openclaw uses the same skill files as other agents

## Session 2

## Goal

Fix metadata inconsistencies in `SKILL.md` flagged by ClawdHub during skill registry review.

## What Changed

- **`skills/cal-cli/SKILL.md`**: Fixed four frontmatter/content issues:
  - `name: cal-cli` → `name: ical` (matches binary name, repo name, and go module)
  - `author: sidv` → `author: BRO3886` (matches GitHub org that owns the repo)
  - `version: "1.0"` → `version: "0.7.0"` (matches latest git tag)
  - `git clone <repo-url>` → `git clone https://github.com/BRO3886/ical` (real URL, no placeholder)

## Key Insights

- ClawdHub (skill registry reviewer) flagged the mismatches between SKILL.md metadata and actual registry entry as a trust signal concern — name/author/version inconsistencies make it harder to verify the skill's provenance
- `name` in SKILL.md frontmatter should always match the binary name (what you actually type in the terminal), not the internal directory name (`cal-cli` was the directory, `ical` is the command)
- `version` in SKILL.md should track the latest release tag, not a separate versioning scheme — this avoids the "skill says 1.0, binary is 0.7.0" confusion
- The placeholder `<repo-url>` in the build-from-source example was a leftover from early development and never got updated — easy miss since it's not tested

## Decisions Made

- Kept the skill directory name as `cal-cli/` (not renamed to `ical/`) since that's the embed path and would require changes across `skills.go`, `internal/skills/`, install paths, etc. The `name:` frontmatter field is what the registry reads, not the directory name.
